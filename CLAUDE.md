# lazy-netlog-viewer — project context for Claude Code

懒猫微服 (LazyCat MicroServer) lpk wrapper for Chromium's
[Net Log Viewer](https://netlog-viewer.appspot.com/) — the official static
visualizer for `chrome://net-export/` NetLog files.

Upstream lives in [catapult/netlog_viewer/](https://chromium.googlesource.com/catapult/+/refs/heads/main/netlog_viewer/).
We do **not** vendor catapult — the repo is huge and we only need a sliver.
Instead, the docker builder stage sparse-clones the parts we need from
chromium.googlesource.com on every image build.

Build/publish/bootstrap logic is **not** in this repo — it lives in
[microlazy-apps/lazycat-ci](https://github.com/microlazy-apps/lazycat-ci).

## Layout

```
lazy-netlog-viewer/
├── vendor/
│   └── netlog-viewer/           docker build context
│       ├── Dockerfile           sparse-clone catapult + nginx:alpine
│       ├── nginx.conf
│       └── .dockerignore
├── lazycat/
│   ├── package.template.yml     committed; version: ${VERSION}
│   ├── lzc-manifest.template.yml committed; image: ${LAZYCAT_IMAGE}
│   ├── lzc-build.yml            committed
│   ├── appstore.yml             submission-only metadata
│   ├── icon.png                 (TODO — drop a 512x512 png here)
│   ├── screenshots/             (TODO — capture from running container)
│   ├── package.yml              .gitignored, generated by build-lpk.sh
│   ├── lzc-manifest.yml         .gitignored, generated by build-lpk.sh
│   └── .gitignore               *.lpk, generated yamls
└── .github/workflows/
    ├── release.yml              thin wrapper, calls lazycat-ci reusables
    └── bootstrap-app.yml        thin wrapper, calls lazycat-ci reusables
```

There is **no `patches/` directory** — we don't patch the upstream. The
Dockerfile + nginx.conf are the only adapters and they live in
`vendor/netlog-viewer/` next to the docker context. `release.yml` still
passes `patches-target: vendor/netlog-viewer` for symmetry with the
lazycat-ci reusable workflow; build-lpk.sh just no-ops when `patches/*`
yields no matches.

## Architecture: pure static nginx

Net Log Viewer is 100% client-side: the user opens it in a browser, picks a
local NetLog `.json`, and the JavaScript parses + visualizes everything in
the browser. Nothing is uploaded.

So the container is just `nginx:1.27-alpine` serving:

- `/` → `index.html` (catapult/netlog_viewer/netlog_viewer/index.html)
- `/components/*` → `third_party/polymer/components/*` (Polymer 1 + webcomponentsjs polyfill)
- `/third_party/*` → catapult `third_party/*` (jszip, polymer-svg-template)
- `/healthz` → `200 ok` (lazycat health check)

No backend, no env vars, no persistent volumes.

## Dev layout vs vulcanized

The official appspot deploy (`netlog-viewer.appspot.com`) serves a single
`vulcanized.html` produced by
`netlog_viewer/netlog_viewer_build/build_for_appengine.py`, which inlines
all HTML imports / JS / CSS via the deprecated `vulcanize` npm tool.

We deliberately **don't vulcanize**. Reasons:

- `vulcanize` is deprecated (last published 2017), depends on old Node, and
  is finicky to install reliably in CI.
- The dev layout works perfectly under the same webcomponentsjs polyfill
  the appspot site uses, just with more HTTP requests on first load.
- Sparse-cloning + plain nginx is simpler and more debuggable than a
  multi-stage build with node 14 + npm + vulcanize + Python.

If we ever care about page-load performance enough to want vulcanize back,
the right move is probably to switch to one of the modern ports of the same
tool (e.g. polymer-bundler) rather than reviving vulcanize.

## Persistent volumes

None. Net Log Viewer has no server-side state.

## Sparse-clone strategy

The Dockerfile builder stage:

```sh
git clone --depth=1 --filter=blob:none --sparse \
  --branch ${CATAPULT_REF} \
  https://chromium.googlesource.com/catapult.git catapult
git -C catapult sparse-checkout init --cone
git -C catapult sparse-checkout set \
  netlog_viewer \
  third_party/polymer \
  third_party/polymer-svg-template \
  third_party/jszip
```

`CATAPULT_REF` is a build-arg defaulting to `main`. Pin it to a specific
revision when you want reproducible builds (e.g. when triaging a regression
introduced upstream).

Sources sparse-cloned:

| Path | Why |
|------|-----|
| `netlog_viewer/netlog_viewer/` | the app — HTML/JS/CSS files |
| `third_party/polymer/components/` | mounted at `/components/` (Polymer 1 + webcomponentsjs) |
| `third_party/jszip/` | served at `/third_party/jszip/jszip.min.js` (zipped NetLogs) |
| `third_party/polymer-svg-template/` | referenced by some Polymer components |

## Release flow (`v*` tag)

```
git tag v0.1.0 && git push --tags
  ↓
release.yml (caller, this repo)
  ├── prep:    strip leading v -> 0.1.0
  ├── build:   uses lazycat-ci/lpk-build.yml@v0.1.0
  │             docker (ubuntu) -> push ghcr (small image: nginx:alpine + static)
  │             build  (xiaomoru) -> copy-image -> render templates ->
  │                                   lzc-cli project build -> artifact:lpk
  └── publish: uses lazycat-ci/publish.yml@v0.1.0
               publish-release (ubuntu) -> GH Release upload .lpk
               publish-appstore (xiaomoru) -> lzc-cli appstore publish
```

Image is small (~30-50 MB) because it's just nginx + a sliver of catapult
sources.

## One-time setup (org-level)

- Org self-hosted runner `xiaomoru` already registered at microlazy-apps level
- Org/repo secrets `LAZYCAT_USERNAME`, `LAZYCAT_PASSWORD` already in place
- ghcr package `microlazy-apps/lazy-netlog-viewer` visibility — defaults to
  PRIVATE on first push, must be flipped to Public at
  https://github.com/orgs/microlazy-apps/packages/container/lazy-netlog-viewer/settings
  before lazycat's `copy-image` can pull it.
- Lazycat developer center registration of `cloud.lazycat.app.netlog-viewer`
  via `bootstrap-app.yml`.

## Updating Net Log Viewer upstream

There is no manual upstream update step — every docker build re-fetches
catapult HEAD via sparse-clone. To freeze a known-good revision, set
`CATAPULT_REF` in the Dockerfile (or thread it through as a workflow input).

## Local lpk build (smoke test)

Requires `lzc-cli`, `envsubst` (gettext), and docker on PATH.

```sh
# one-time
git clone https://github.com/microlazy-apps/lazycat-ci ~/lazycat-ci

# in this repo:
bash ~/lazycat-ci/scripts/build-lpk.sh \
  --image cloud.lazycat.app.netlog-viewer:0.0.0-dev \
  --version 0.0.0-dev \
  --docker-context ./vendor/netlog-viewer \
  --patches-target vendor/netlog-viewer
# → lazycat/cloud.lazycat.app.netlog-viewer-0.0.0-dev.lpk
```

## Versioning notes

- Upstream catapult has no semantic version — it's the Chromium tooling
  monorepo, versioned by commit.
- This wrapper picks its own semver starting at `v0.1.0`.
- Bump version on Dockerfile / nginx.conf changes, or when re-pinning
  `CATAPULT_REF`.
- Lazycat appstore enforces strict monotonic semver.
